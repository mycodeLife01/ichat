# 前端 UI/UX 重构设计

日期：2026-05-23

## 目标

把当前用于验证后端能力的原生 JavaScript 测试前端，重新设计为安静、专注、成熟的 AI 聊天产品界面。首期聚焦现有能力的用户体验，不新增依赖后端扩展的产品功能。

本设计是前端重构的第一阶段，只定义 UI/UX 蓝图与体验验收标准。React、Vite、TypeScript、请求层、状态管理与自动化测试框架等工程选择，将在下一阶段单独设计。

## 背景与约束

当前 `frontend/` 由 FastAPI 以同源静态资源方式挂载，虽然实现为原生 JavaScript + Tailwind CDN，但已承载以下真实产品行为：

- 登录、注册、刷新认证与退出登录。
- 新建草稿对话、首次成功回答后的激活与自动标题更新。
- 历史对话切换、重命名与删除。
- SSE 流式回答、`after_seq` 重放与中断后的继续接管。
- 停止生成、失败和取消后的部分内容展示。
- 用户消息编辑并重新生成、助手回答重新生成。
- 思考过程的实时展示、自动收起与历史回看。

新 UI 必须保留这些行为语义，不得把后端现有的草稿激活、流式重放、归档重生成或认证清理机制简化掉。

## 已确认的产品方向

| 决策点 | 选择 |
|--------|------|
| 首份设计范围 | UI/UX 蓝图；工程架构另行设计 |
| 首期功能范围 | 重塑现有核心体验，不设计未实现业务 |
| 产品气质 | 安静、专注、成熟 |
| 界面语言 | 中文优先，包括错误反馈与应用提示 |
| 设备优先级 | 桌面端与手机端都是一等体验 |
| 登录/注册权重 | 认真设计但从简 |
| 工作区布局 | 专注工作台：桌面常驻侧栏，手机抽屉 |
| 消息操作 | 默认纯净，操作按需出现 |
| 空白新对话页 | 一句克制欢迎语 + 输入框 |

### 文案约定变更

仓库既有说明要求用户可见错误消息和应用级提示使用英文。本次新前端设计明确覆盖这一约定：**前端用户可见文案统一使用自然、简洁的中文**，包括操作按钮、加载状态、错误反馈、取消反馈和认证提示。

实现阶段应同步更新仓库级 agent 指引或相关约定文档，避免后续贡献继续引入中英文混杂文案。后端直接返回的英文 `detail` 若会展示到 UI，应由前端映射为适合用户理解的中文反馈，或在后续 API 文案整治任务中统一处理。

## 范围

### 首期包含

- 登录与注册入口的视觉和交互重塑。
- 聊天工作区的信息结构、桌面与手机布局。
- 新对话空白状态、输入与发送体验。
- 历史对话导航、标题加载、重命名与删除体验。
- 用户消息、助手消息与 Markdown 阅读排版。
- 思考过程、流式输出、停止生成与终态反馈。
- 复制、编辑并重新生成、重新生成回答的交互方式。
- 页面刷新、重连、认证过期和失败状态的呈现。
- 键盘、触屏和基础无障碍要求。

### 首期不包含

- 深色模式。
- 模型切换、参数控制或思考强度控制。
- 文件或图片上传。
- 对话搜索、标签、文件夹或归档管理。
- 对话分支可视化、分享或导出。
- 设置中心与用户资料管理。
- 空白页的提示模板卡片。
- 任何需要新增后端 API 的能力。

## 信息结构

首期只有两个一级界面：

1. **认证入口**：登录或注册后进入聊天工作区。
2. **聊天工作区**：登录用户的主界面，承载新建、历史浏览和全部对话操作。

聊天工作区分为三层：

| 层级 | 职责 | 主要内容 |
|------|------|----------|
| 历史导航层 | 查找和管理已建立的对话 | 产品标识、新建对话、已激活历史列表、账号信息、退出 |
| 对话阅读层 | 专注阅读当前会话 | 当前标题、消息流、空白欢迎状态、移动端历史入口 |
| 输入与运行层 | 发起请求和控制生成 | 多行输入框、发送/停止操作、轻量运行状态提示 |

草稿对话遵循现有产品语义：创建后可以在当前阅读层中使用和恢复，但在首次助手回答成功前不进入历史列表。

## 布局设计

### 桌面端

桌面端采用“专注工作台”结构：

- 左侧为常驻窄侧栏，承载历史导航层。
- 右侧为主要内容区域，承载阅读层和输入层。
- 侧栏使用低对比度背景和轻分隔，不与回答正文竞争注意力。
- 消息阅读列限定舒适的最大宽度，避免宽屏长行降低阅读效率。
- 输入框与消息阅读列对齐，形成稳定的视觉轴线。
- 顶部标题栏保持简洁，仅呈现当前标题和必要交互。

### 手机端

手机端保持完整能力而不是缩减版体验：

- 历史侧栏改为从边缘打开的抽屉，主界面顶部提供明确入口。
- 抽屉打开时具备遮罩和可发现的关闭方式。
- 消息区域接近全宽但保留适合拇指与阅读的边距。
- 输入区靠近屏幕底部，并考虑安全区与软键盘遮挡。
- 所有消息操作提供触控可用入口，不依赖桌面端的 hover 机制。

## 认证入口

登录与注册入口保持简洁而可信，不承担产品介绍页面的任务：

- 使用居中的单栏表单容器，呈现 `iChat` 标识和登录/注册切换。
- 移除“测试客户端”等内部开发语义。
- 字段标签、校验信息、提交状态均采用中文。
- 错误信息就近显示在相应字段附近；仅对跨表单的结果反馈使用全局提示。
- 手机输入字号和控件触控区域符合移动端使用要求，避免聚焦时页面异常缩放。

## 新对话与输入体验

### 空白状态

用户进入新对话时，主区域仅展示：

- 克制的欢迎语：“今天想聊些什么？”
- 底部主输入框。

首期不展示示例问题、能力介绍或快捷提示卡片。界面应让用户自然把注意力放在输入行为上。

### 输入规则

- 输入框占据底部主要操作位置，首期 placeholder 为“向 iChat 提问”。
- 支持内容增长时的多行展开，并设置合理最大高度，超出后内部滚动。
- 桌面端使用 `Enter` 发送，`Shift+Enter` 换行。
- 输入法组合输入期间不得因 `Enter` 误发送。
- 用户无需先建立或命名对话；发送第一条消息时在后台创建或使用草稿对话。

## 消息阅读与内容呈现

### 用户消息

- 采用浅色轻背景气泡，靠右呈现。
- 视觉上清楚区分于助手正文，但不过度强调。
- 编辑状态沿用同一气泡视觉语言，就地切换为多行编辑框。

### 助手消息

- 不采用厚重气泡；以靠左的正文排版呈现。
- 长回答应具备舒适的段落间距和行宽。
- Markdown 的标题、段落、列表、引用、链接、行内代码和代码块需要具备稳定层级。
- 代码块在首期强调可读性，不扩展复杂工具栏或未确认的附加操作。

### 自动滚动

- 流式回答开始时立即创建助手占位区域。
- 用户位于消息底部附近时，新增内容随流式生成平滑跟随。
- 用户已向上阅读历史时，新增内容不得强制把视口拉回底部。

## 思考过程

思考过程是回答的次要信息层，而非与正文同等权重的内容：

- 助手消息具有思考内容时，在最终正文上方展示可折叠区域。
- 思考正在生成时默认展开，标签使用“思考中…”。
- 正文开始输出或回答结束后，自动收起并显示“已思考”。
- 用户可主动重新展开历史消息中的思考内容。
- 使用较小字号、弱对比色与轻量分隔线，保持可读但不抢夺正文关注度。
- 刷新或断线恢复时，已生成的思考内容应与正文一起恢复，再继续接收后续流。

## 消息操作

消息操作遵循“默认纯净，按需出现”的原则：

### 桌面端

- hover 或键盘聚焦消息时显示适用操作。
- 用户消息支持复制、编辑并重新生成。
- 助手消息支持复制、重新生成。

### 手机端

- 每条消息提供低强调度的“更多操作”按钮；点击后打开底部操作面板。
- 底部操作面板提供与桌面端一致的适用能力，不将核心能力隐藏为桌面特权。

### 运行中的限制

- 流式生成期间仍允许复制已有内容。
- 编辑并重新生成、重新生成回答在当前 run 未结束时不可执行。
- 禁用时应提供清楚的中文原因，例如“请先停止当前生成”。

### 编辑体验

- 用户编辑消息时在原消息位置切换编辑状态。
- 编辑区复用用户气泡的视觉风格。
- 操作按钮为“发送”和“取消”。
- 提交后进入新的流式回答流程，归档后的旧消息不应继续留在界面中。

## 历史导航与标题

### 新建与草稿

- “新建对话”进入一个新的空白会话状态。
- 未发送消息或尚未成功完成回答的草稿，不进入历史列表。
- 草稿在页面刷新后仍可恢复当前会话内容或进行中的流式生成。

### 自动标题

- 首次回答成功后，对话被激活并加入历史列表。
- 自动标题尚未生成完成时，对应历史行展示稳定的加载占位，不短暂闪现默认标题造成跳动。
- 标题生成失败时回退展示“新对话”，不影响用户继续对话。

### 历史操作

- 已展示在列表中的对话支持重命名和删除。
- 侧栏中的管理操作在桌面端按需出现，在手机端以触控友好的方式触发。
- 删除需要清楚确认，避免误操作。

## 生成、取消与终态反馈

### 流式生成

- 发送成功后，用户消息立即显示，助手占位紧接出现。
- 流式正文末尾以低干扰生成标记表示输出仍在继续。
- 输入区主操作由“发送”切换为“停止生成”。
- 输入区附近显示轻量说明“正在生成，可停止”。

### 请求停止

- 用户点击“停止生成”后，按钮立即进入不可重复操作的停止中状态。
- 停止中提示固定为“正在停止…”。
- 直到服务端到达取消终态前，不伪装为已经停止。
- 已生成的部分正文和思考内容继续保留，终止后显示轻量“已停止”标记。

后端和事件流中的 `cancelled` 仍表示技术终态；产品界面统一将该用户触发的动作表述为“停止生成”与“已停止”。

### 失败

- 流式生成失败时保留已经到达的可见内容，并标识失败终态。
- 错误反馈应短、具体、使用中文，不向普通用户暴露不必要的内部错误细节。
- 可恢复的操作应给出下一步，例如允许用户重新发送或重新生成。

## 恢复与认证状态

### 流式恢复

- 刷新页面或重新进入进行中的会话时，先呈现服务端已持久化的正文和思考过程。
- 之后基于现有 SSE cursor 机制继续接收新增内容。
- 用户不应看到回答从空白重新生成，或已出现内容重复追加。

### 认证过期与账户切换

- access token 失效但可刷新时，界面尽量无感继续当前操作。
- refresh 失败时，清理当前对话选择、草稿引用和流式状态，并返回认证入口。
- 新身份登录后不得看到前一身份留存在内存或页面中的私有内容。

### 对话不可访问

- 当前恢复目标已删除或不属于当前用户时，清理失效选择并回到可用空白状态。
- 不在正常的账户切换或过期恢复场景中展示生硬的资源错误提示。

## 视觉系统方向

首期只定义足以指导 UI 落地的视觉原则，不在本设计中确定组件库或 CSS 技术实现：

- 默认使用高质量浅色主题。
- 大面积背景采用中性浅色，正文使用高可读性的深色。
- 侧栏、边框、占位与次要状态使用低对比层级。
- 强调色仅用于焦点、当前状态和必要反馈，避免大面积抢眼品牌色。
- 层级主要依靠留白、字重、排版、轻边框和克制圆角建立。
- 避免重阴影、大面积渐变和装饰性欢迎插画。
- 动效服务于理解：抽屉开合、流式指示、标题加载和折叠过渡应短促稳定，不制造干扰。

## 基础无障碍与交互质量

- 所有按钮和图标操作具备明确的可访问名称。
- 键盘用户能识别当前焦点，并完成登录、发送、停止和消息操作。
- 思考区域使用可表达展开状态的控件。
- 移动抽屉可通过清晰按钮和遮罩关闭；后续实现需考虑焦点管理。
- 提示和状态不应只依赖颜色传达。
- 移动端主要点击目标应足够大，输入区不受软键盘和安全区明显干扰。

## 关键状态矩阵

| 状态 | 主区域 | 输入区 | 历史侧栏 |
|------|--------|--------|----------|
| 未登录 | 认证入口 | 不适用 | 不适用 |
| 新空白对话 | 欢迎语 + 空内容 | 可发送 | 不添加草稿行 |
| 加载历史对话 | 内容加载反馈 | 暂不可产生冲突操作 | 选中项保持稳定 |
| 生成中 | 用户消息 + 助手流式占位；思考可展开 | 停止操作 + 生成提示 | 当前已激活对话保持选中 |
| 停止中 | 已有流式内容保留 | 停止中、不可重复点击 | 不特殊跳动 |
| 已取消 | 部分内容保留并标注“已停止” | 恢复可发送 | 未激活草稿仍不显示 |
| 生成失败 | 部分内容保留并标注失败 | 恢复可操作 | 不新增失败草稿 |
| 首次成功、标题生成中 | 正常最终回答 | 可继续输入 | 新行显示标题占位 |
| 首次成功、标题完成 | 正常最终回答 | 可继续输入 | 新行显示自动标题 |
| SSE 恢复中 | 先回填既有内容，再继续增长 | 根据 active run 展示停止能力 | 状态不重复创建 |
| 认证失效 | 返回登录入口 | 不适用 | 清空私有状态 |

## 后续工程设计输入

下一阶段 React 工程重构设计必须以本规格为约束，尤其需要覆盖：

- 认证态、会话选择态、草稿态、标题 pending 态和 active run 态的清晰建模。
- SSE 事件重放、流式增量更新和页面恢复的可靠实现。
- 桌面/移动布局与触控操作的组件边界。
- Markdown 安全渲染和消息操作的测试策略。
- 中文用户反馈文案的统一承载方式。
- FastAPI 同源部署与前端构建产物的集成方式。

## 验收标准

### 体验流程

- 桌面端与手机端均能完成注册、登录、开始新对话、发送消息、流式阅读、停止生成、查看思考过程、复制、编辑并重新生成、重新生成回答、历史重命名与删除、退出登录。
- 新对话无需手动命名或预先进入历史列表；首次成功回复后的激活和标题加载体验稳定。
- 页面刷新后可以继续当前草稿和进行中的回答，正文与思考过程不会明显丢失或重复。

### 界面质量

- 主要界面文案和用户反馈统一为中文，无无意中英文混杂。
- 桌面布局具备清楚的历史/阅读/输入层次；手机布局的侧栏与消息操作完整可用。
- 空白、加载、生成、停止中、已停止、失败、标题生成和恢复状态均有明确且克制的表达。
- 消息长文与 Markdown 内容可舒适阅读，操作入口不会长期压过正文。

### 行为保真

- 不破坏后端现有的 SSE 回放与恢复语义。
- 不破坏草稿对话首次成功后才激活的语义。
- 不破坏自动标题异步生成与失败回退的语义。
- 不破坏编辑/重新生成引发的消息归档结果。
- 不破坏退出或认证失效后的用户状态隔离。
